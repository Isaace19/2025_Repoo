Introduction

Dynamic shared objects and dynamically linked libraries are ways to import code or data into a program while it is running. The term dynamic refers to the fact that these libraries are imported during runtime instead of a typical archive or static library, where the code is imported during the linking stage.

Dynamic Shared Objects

A dynamic shared object is a fancy term for code (CPU instructions) and data that can be loaded at runtime. This can be a little bit dangerous since your compiled program cannot vouch for the integrity of the plugin (DSO). Usually, hashes are computed to determine the integrity of the plugin. However, if the integrity of a plugin is not verified, you could potentially run malicious code.

To use dynamic library functions, include <dlfcn.h>. The library -ldl will also need to be specified when compiling.
Function Description

void *dlopen(const char *filename, int flags)

Opens a dynamic library specified by filename and returns an opaque pointer called a handle.

int dlclose(void *handle)

Closes a dynamic library that was given to you as handle.

void *dlsym(void *handle, const char *symbol)

Looks up symbol in the dynamic library and returns a pointer to where it loaded it.

char *dlerror(void)

Returns a string that gives a human-readable reason for the latest error.

Dynamic Shared Object Code

Usually, we write starting at int main, but remember, this is where the compiler puts the starting point for your code. For DSOs, we add symbols to a shared object (.so) file. Then, we find those symbols at run time.

You can compile DSOs by using the -shared switch.

~> gcc -Wall -O0 -g -shared -o libfmadd.so fmadd.c

This will create a shared object (dynamic library) called libfmadd.so. This is the file you will be able to dynamically import into your code.

Function Pointers

Recall that the term function simply refers to where the instructions are stored in memory, usually in the .text section. So, the void * pointer we get back is a memory address where we can find CPU instructions. In fact, we can see the instruction via the DSO by examining the memory of fcnptr.

unsigned char *insn = (unsigned char *)fcnptr;

printf("%2x %2x %2x\n", insn[1], insn[2], insn[3]);

On my computer, the code above prints 48 89 e5, which is the instruction mov rbp, rsp, which takes three bytes to encode. This sets up the stack frame for the fmadd function. I skipped insn[0], which is the push instruction to save the rbp register, called the base pointer, aka the frame pointer.

Type Definitions

Function pointers can be type defined using typedef, but the syntax is slightly weird as the name is specified in the function wrapping.

If you look at plugin.h, you will see three type definitions.

plugin.h - partial
typedef void (*INIT)(void);
typedef PluginResult (*SETTING)(const char name[], const char value[]);
typedef PluginResult (*TRANSFORM)(char *data, int *size);

We can now create a structure to hold these function pointers, if needed.

typedef struct {
    INIT init;
    SETTING setting;
    TRANSFORM transform;
} PluginFunctions;

This creates a structure with three function pointers using the type definitions. It makes the syntax much clearer.

Since the pointer is in the typedef itself, do NOT specify it in the structure. Otherwise, you now have a double pointer. In the example above INIT, SETTING, and TRANSFORM specify pointers.

Potential Uses

A dynamic shared object, or in Windows, a dynamic linked library (DLL), can be used to plug in code at run time. We dont have to recompile the executable, but instead, if we need to change something, we can change the code in the plugin, and only recompile a small portion of the entire program.

Lab Assignment

You will create a program that allows third-party plugins via dynamic shared objects in main.c. To be able to support DSOs, your code will have a standard plug-in structure that other developers can use to create plugins for your program.

You will also create three plugins to work with your loader: flip.c, substitute.c, and reverse.c. A fourth plugin skip.c has been written for you as a demonstration.

Your main.c will be tested with other plugins, and your plugins will be tested against a solution loader.

Loader Program

Your main.c will be the loader program. Its job is to take input from the user via the command line arguments. The last command line argument will be the name of a file.

To specify a plugin, the user will use the -p switch followed by the name of the plugin. The name will not be the full name of the plugin. Instead, your program will automatically add libXXYY.so where XXYY is whats given to your -p argument. For example, -p flip will open a plugin called ./libflip.so, and -p libflip.so will open a plugin called ./liblibflip.so.so. In other words, prepend ./lib and append .so to whatever is given to your -p switch.

Recall that dlopen uses a DL cache if a relative path is passed to it. This is why you need to prepend ./ to tell dlopen to only look in the current directory.

Multiple plugins can be specified with multiple -p switches. The order must be preserved. In other words, the first plugin specified must execute first, followed by the next, and so on.

You can give settings to your plugin by using the -s switch. The -s switch is followed by a name=value sequence or a name sequence. For the latter, with just a name, the value will be the empty string "". Multiple settings can be given by using multiple -s switches. For example, -p plugin -s setting1=value1 -s setting2 will cause libplugin.so to be loaded, followed by two calls to the setting function. The first will have the name parameter setting1 and the value parameter value1, and the second will have the name parameter setting2 and the value parameter "" (empty string).

Do not use heap memory for the loader program. It is unnecessary. Instead, run the plugins as you parse the arguments. Since the data is changed in place (in memory), it can be passed as a pointer to each plugin.
Since you need the file to run any plugin, you can check the last command line argument by opening it and seeing if it opens read-only. You will NOT be writing to the file.

File Handling

You will need to use mmap to map the file into memory as the data and size pointers which are passed to the plugins. Since the file will be opened using O_RDONLY, you must ensure that you map the file into memory for the program only (i.e., writes do not write back to the file). However, since the memory will change within the plugins, you need to map it given the protections PROT_READ and PROT_WRITE.

Do not forget to error check mmap, and when youre finished with the memory to munmap it.
Recall that you can close the file descriptor immediately following a successful mmap.

Loading System

When your loader gets a plugin, it will load the plugin by using dlopen. It will then find three functions: init, setting, and transform. Both init and setting are optional, so those returning NULL with a dlsym call is not an error.

  • The first call will be to init, if it exists. The init function in the plugins will help ensure two or more loads of the same plugin have a deterministic starting point.

  • The second will be any setting specified with the switch -s. If there is no setting function in the plugin, but a -s switch has been specified, output the error message: "ERROR: ./libxxyy.so does not take settings.", where ./libxxyy.so is the name of the plugin given by -p.

  • The third will be the transform with the memory pointer and size pointer passed. If transform is NULL, this is an error, and your loader will print "ERROR: unable to find 'transform' function in plugin ./libxxyy.so." (replace ./libxxyy.so with the name of the plugin) and quit.

  • After all of the above has occurred, close the plugin by calling dlclose. Plugins will not remain resident even if the same plugin is specified by another -p.

  • After all plugins execute, and barring any error which causes the loader to quit, print the data pointer followed by printing a newline character to standard out.

PluginResult

An enumeration called PluginResult is the return value for both transform and setting. The init function takes no arguments and returns nothing.

The PluginResult has different meanings depending on the function.

PluginResult for Transform Function

For the transform function, the PluginResult has the following meanings:

  • PR_SUCCESS - a plugin result of PR_SUCCESS tells the loader program that the transformation succeeded. There is nothing your program needs to do in response to this.

  • PR_FAILED - a plugin result of PR_FAILED tells the loader program that the transformation failed. In this case, your program will print an error message: ERROR: plugin './libsoln.so' failed.. Obviously, replace ./libsoln.so with the name of the plugin that failed. Your loader will move on to the next plugin after displaying the error message.

  • PR_STOP - a plugin result of PR_STOP tells the loader program a "fatal error" has occurred, and it should stop executing.

PluginResult for Setting Function

For the setting function, the PluginResult has the following meanings:

  • PR_SUCCESS - a plugin result of PR_SUCCESS tells the loader program that the setting succeeded. There is nothing your program needs to do in response to this.

  • PR_FAILED - a plugin result of PR_FAILED tells the loader program that the setting failed. This could mean that either the name is not recognized, or the value is not appropriate for the given setting. If your loader gets a PR_FAILED result, it will print "ERROR: invalid plugin argument for ./libxxy.so: somename". Replace ./libxxy.so with the name of the plugin, and replace somename with the name of the setting given by -s.

  • PR_STOP - a plugin result of PR_STOP tells the loader program that the setting error was handled by the plugin itself. The loader program must be silent.

Regardless of whether or not your program prints an error message, for the setting function, a PR_FAILED or PR_STOP will quit the entire program.

Plugins

You will write four (4) plugins: flip, reverse, substitute, and uppercase. The plugin skip.c has already been written for you to demonstrate how your plugins should be structured.

For all plugins, if a setting is not recognized, return PR_FAILED. Also, all default settings must be set each time init is called.

uppercase

The uppercase plugin will take all letters a-z and make them A-Z. In other words, it will make them uppercase. You can use the <ctype.h> header with the functions toupper to change a single character from lowercase to uppercase.

There will be one setting for this plugin called stride. The stride is how far your for loop will "stride" after each iteration. For example, if I set the stride to 4, after the first character, the 4th character gets uppercase, then the 8th, and so on.

For example, if I have abcdefghijklmnop and I set the stride to 4, I get AbcdEfghIjklMnop. You can see that every fourth letter is uppercase.

By default, the stride is 1, meaning every character will be uppercase.

The stride setting must be an integer > 0. If it is not, return PR_FAILED.

substitute

The substitute plugin will take two settings upper and lower. These are strings of 26 characters. By default upper is abcdefghijklmnopqrstuvwxyz and lower is ABCDEFGHIJKLMNOPQRSTUVWXYZ. If left default, the substitute plugin will change all uppercase to lowercase and all lowercase to uppercase.

The upper string tells your plugin what to do when it encounters an uppercase character. For example, if I get C, that is index 2 in the upper string, which corresponds with c, so C is substituted with c. The lower string is analogous to the upper string, except it tells your plugin what to do when it encounters a lowercase character.

You can use <ctype.h> which contains isupper and islower. Any non-letter characters are left as they were and are not modified.

When setting upper and lower, the strings given to these settings must be less than 26 characters. If they are not, return PR_FAILED. If the strings are not exactly 26 characters, copy starting at the first index until you run out of characters. For example, if I set upper to zxy, then the upper substitution string will be zxydefghijklmnopqrstuvwxyz, in other words, only the first three characters are modified, and the rest of the string is untouched.

reverse

The reverse plugin reverses the data character-by-character. For example, Hello World! would become !dlroW olleH. In other words, the last character in the file becomes the first. The next-to-last-character becomes the second, and so forth. This includes any non-printing characters, such as a new line.

There are no settings for the reverse plugin.

Make sure you omit the init and setting functions in the reverse plugin.

flip

The flip plugin flips data line-by-line vertically (top-to-bottom).

Before (unmodified)
Hello, world!
That's what I said!
Ok, in that case, goodbye!
Vertical Flip (after)
Ok, in that case, goodbye!
That's what I said!
Hello, world!

There are no init or setting functions in this plugin.

Compiling

Your loader and plugins must compile even if they arent correct. The grade script cannot continue executing if anything fails to compile or if theres a memory leak in your loader or plugins.

  • Your loader will be compiled with gcc -std=gnu11 -ldl.

  • Your plugins will be compiled with gcc -std=gnu11 -fPIC -shared.
